## Raw Request

- Plan the order process from placement until parcel delivery.
- Include shipping, parcel tracking, parcel return, COD, payment handling, courier sync, customer complaints, and standard naming conventions for order, invoice, and shipment numbers.
- New database tables and columns may be needed.

## Objective

Design the canonical backend workflow and schema for orders after checkout. This includes identifiers, status transitions, shipment records, courier events, return flow, complaint flow, COD tracking, and the minimal audit trail required for admin and mobile operations to act on an order confidently.

## Acceptance Criteria

1. The task defines the target lifecycle for `order`, `shipment`, `payment`, `return`, and `complaint` records.
2. The task recommends naming conventions and uniqueness rules for platform-generated identifiers.
3. The task defines which data belongs on `orders` versus new related tables.
4. The task identifies the minimum columns, indexes, and foreign keys needed for shipment and courier tracking.
5. The task defines event history requirements so status changes are traceable.
6. The task identifies how multi-package shipments and partial shipment scenarios should be represented, even if deferred for phase two.

## Likely Implementation Areas

- `orders` table and related Eloquent model updates
- New `shipments` table
- New `shipment_events` or `courier_tracking_events` table
- New `returns` and `return_items` tables if itemized returns are required
- New `order_complaints` or `support_cases` table if complaints are operationally linked to orders
- New order/payment/shipment DTOs and admin/mobile payload contracts
- Status transition rules and audit trail support

## Risks Or Open Questions

- Suggested identifier conventions:
- `orders.public_id`: keep existing public customer-facing `ORD-...` format unless already fixed in production and unsuitable to change.
- `invoices.invoice_number`: `INV-YYYY-#####` or `INV-YYYYMM-#####`.
- `shipments.shipment_number`: platform-generated `SHP-YYYY-#####`.
- `shipments.tracking_number`: courier-provided AWB or tracking code, not generated by the platform unless courier integration is absent.
- It is unclear whether one order can generate multiple invoices.
- It is unclear whether the courier tracking number should exist before dispatch or only after courier handoff.
- COD may require extra fields such as `cod_amount`, `cod_currency`, `cod_collected_at`, `cod_remitted_at`, and courier remittance reference.
- Complaint workflow may need severity, resolution type, SLA target, and replacement/refund linkage.
- Return workflow may need both `return_status` and a separate `return_shipment_tracking_number`.
