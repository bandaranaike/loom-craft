# Task: Order Details And Tracking

## Goal

Upgrade the order details experience so each order has a durable platform-generated public identifier, a proper order
details page, accurate currency display, and a visual status timeline that reflects the real lifecycle of the order.

## Current Context

- The app already has order pages at `/orders/{order}` and an order confirmation page.
- Those routes currently use the numeric database ID.
- The task list calls for a unique string-based order identifier, likely longer than a UUID, for public-facing usage.
- The order view already exists, but the task list indicates the current page still needs stronger order-details
  behavior:
    - stable public order identifier
    - status progress visualization
    - correct currency display
- The task list also notes that prices are currently showing as dollars in the order view even though currency is
  dynamic.

## Problem To Solve

1. Numeric order IDs are weak as public-facing identifiers.
2. The order details page needs a stable lookup key suitable for customer-facing URLs.
3. Order progress is not currently represented as a customer-friendly lifecycle.
4. Currency rendering on the order page is inconsistent with the stored order/payment currency.

## Scope

### In Scope

1. Introduce a platform-generated public order identifier.
2. Update order detail routing to use that identifier for the customer-facing page.
3. Improve the order details page at `/orders/{order_id}`.
4. Add a visual order progress component.
5. Fix order-view currency display to use the actual order/payment currency.

### Out Of Scope For This Task

1. Full shipment tracking integration with carriers.
2. Customer self-service cancellation/refund flows unless needed by the chosen status model.
3. Replacing the internal numeric primary key.

## Recommended Product Direction

1. Keep the numeric primary key internally.
2. Add a separate public order key on `orders` for URLs and customer references.
3. Use a long, non-sequential identifier generated by the platform.
4. Treat the order details page as the canonical order tracking page.
5. Render status as a clear sequence such as:

- Order placed
- Payment done
- Order confirmed
- Order delivered
- Order canceled

## Backend Work

### Public Order Identifier

1. Add a unique public identifier column to `orders`.
2. Generate it automatically when the order is created.
3. Keep it immutable after creation.
4. Update route model resolution or controller lookup so the customer-facing order page uses the public key instead of
   the numeric ID.

### Order Details Data

1. Extend order DTOs/resources so the page receives the public order ID and the correct currency fields.
2. Ensure the page uses the persisted order/payment currency instead of any hard-coded dollar formatting.
3. Confirm whether totals should use `orders.currency`, `payments.currency`, or both depending on the display context.

### Status Model

1. Define how existing stored statuses map to the visual progress steps.
2. Decide how canceled orders should interrupt or replace the normal progress flow.
3. Keep the UI mapping centralized rather than duplicating status logic in multiple pages.

## Frontend Work

### Order Details Page

1. Upgrade the order page at `/orders/{order_id}` to be the main tracking/details view.
2. Show the public order identifier prominently.
3. Show core details clearly:

- order date
- payment method
- payment status
- order status
- amount totals
- shipping / billing context as appropriate

### Status Progress UI

1. Add a visual progress/timeline component for order lifecycle status.
2. Handle terminal canceled state clearly.
3. Ensure the status UI works on desktop and mobile.

### Currency Display

1. Fix the order page so prices render in the correct currency.
2. Ensure every total/line item/summary uses the correct symbol and code.
3. Avoid assuming USD anywhere in the order details view.

## Data Model Considerations

1. Public order ID should be unique and indexed.
2. The chosen identifier should be long enough to avoid guessability concerns.
3. If guest order access is expanded later, this public key becomes even more important.

## Testing Requirements

1. Orders receive a unique public identifier when created.
2. The order details route resolves by public order identifier.
3. The numeric internal ID is no longer required in customer-facing order URLs.
4. The order details payload includes the correct currency fields.
5. The order page renders the correct currency for totals and line items.
6. The status timeline reflects paid, pending, delivered, and canceled states correctly.

## Acceptance Criteria

1. Customer-facing order URLs use a platform-generated unique string.
2. The order details page works through that identifier.
3. The page shows a clear visual order-progress state.
4. Prices and totals display in the correct currency.
5. The page is suitable as the primary order reference page for customers.

## Open Decisions

1. What public order key format should the platform use: ULID, UUIDv7, prefixed random token, or another long opaque
   string?
2. Should canceled orders remain on the same timeline with a terminal state, or use a distinct canceled summary card?
3. Should the order details page remain auth-protected only, or should the public order key later support guest lookup
   as well?
